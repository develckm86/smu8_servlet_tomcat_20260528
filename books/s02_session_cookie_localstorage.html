1. 이전 수업 복습

이전 수업에서는 브라우저와 서버가 요청과 응답으로 통신한다는 것을 배웠다.

브라우저→ request→ Tomcat→ Servlet / JSP→ response→ 브라우저 화면

또한 Servlet에서 JSP로 이동하는 두 가지 방법을 배웠다.

구분forwardredirect
이동 방식서버 내부 이동브라우저에게 새 요청을 시킴
주소창바뀌지 않음바뀜
request 데이터유지됨새 요청이므로 사라짐
주 사용 예실패 메시지, 조회 결과 전달로그인 성공, 글 등록 후 목록 이동
이번 수업의 출발점은 redirect 후 request가 사라진다는 점이다. 그런데 로그인 성공 후에는 페이지가 바뀌어도 로그인 상태가 유지되어야 한다. 이 문제를 해결하는 대표적인 방법이 session이다.

2. HTTP는 기본적으로 이전 요청을 기억하지 않는다

브라우저가 서버에 요청을 보내면 서버는 응답을 돌려준다. 그런데 다음 요청이 들어왔을 때 서버는 “아까 그 사용자가 다시 왔다”는 사실을 자동으로 알지 못한다. 이런 성질을 보통 stateless, 즉 무상태라고 부른다.

요청 1

사용자가 로그인 요청을 보낸다.

POST /login
id=admin&pw=1234

요청 2

사용자가 홈 화면을 요청한다.

GET /home.jsp

서버 입장에서는 요청 1과 요청 2가 서로 다른 요청이다. 따라서 로그인 상태를 유지하려면 별도의 저장 방법이 필요하다.

상태 유지가 필요한 예

  • 로그인 후 다른 페이지로 이동해도 로그인 상태 유지
  • 로그인 실패 시 에러 메시지 보여주기
  • 아이디 저장 체크박스
  • 장바구니 또는 최근 본 상품
  • 다크 모드, 글자 크기, 닉네임 같은 화면 설정 저장
상태 유지의 핵심 질문은 하나다. 이 값을 서버가 기억해야 하는가, 브라우저가 기억해도 되는가?

수업 초반에는 이렇게 기억하면 된다. 로그인 상태는 session, 브라우저가 서버에 자동으로 보내도 되는 작은 값은 cookie, 서버가 몰라도 되는 화면 설정은 localStorage가 어울린다.

4. Session

4-1. Session이란?

Session은 서버가 사용자별로 데이터를 저장하는 공간이다. 로그인 성공 후 서버가 세션에 로그인 정보를 넣어두면, 다음 요청에서도 같은 사용자인지 확인할 수 있다.

로그인 성공→ session에 loginId 저장→ redirect→ 다음 요청에서 session 확인

Servlet API의 HttpSession은 여러 페이지 요청 또는 여러 방문에 걸쳐 사용자를 식별하고 사용자 정보를 저장하는 방법을 제공한다. Tomcat 같은 Servlet Container는 보통 세션 ID를 쿠키로 관리하며, 쿠키를 사용할 수 없는 경우 URL rewriting 같은 방법도 사용할 수 있다.

4-2. Session 기본 메서드

메서드의미예시
request.getSession()세션을 가져온다. 없으면 새로 만든다.로그인 성공 시
request.getSession(false)기존 세션만 가져온다. 없으면 null로그인 검사, 로그아웃
session.setAttribute(name, value)세션에 값 저장로그인 아이디 저장
session.getAttribute(name)세션 값 읽기로그인 여부 확인
session.removeAttribute(name)특정 값만 삭제일부 상태 초기화
session.invalidate()세션 전체 종료로그아웃
session.setMaxInactiveInterval(seconds)세션 타임아웃 설정30분 후 만료

4-3. request와 session 차이

구분requestsession
저장 범위한 번의 요청한 사용자 접속 상태
forward 후유지유지
redirect 후사라짐유지 가능
예시로그인 실패 메시지로그인 사용자 정보
로그인 성공은 보통 session에 저장하고 redirect한다. 로그인 실패는 에러 메시지를 request에 저장하고 forward한다.

5. Cookie

5-1. Cookie란?

Cookie는 서버가 브라우저에게 저장하라고 보내는 작은 데이터이다. 브라우저는 조건이 맞으면 다음 요청부터 그 쿠키를 다시 서버로 보낸다.

서버response.addCookie() 브라우저 저장다음 요청 request.getCookies()

Servlet의 Cookie 클래스는 Servlet이 브라우저로 보내고 브라우저가 저장한 뒤 나중에 서버로 다시 보내는 작은 정보를 표현한다. 서버는 response.addCookie()로 쿠키를 보내고, 이후 요청에서는 request.getCookies()로 읽을 수 있다.

5-2. Cookie 기본 메서드

메서드의미
new Cookie(name, value)쿠키 생성
cookie.setMaxAge(seconds)쿠키 유지 시간 설정. 0이면 삭제
cookie.setPath(path)쿠키가 전송될 경로 설정
cookie.setHttpOnly(true)JavaScript에서 쿠키 접근을 막음
cookie.setSecure(true)HTTPS에서만 쿠키 전송
response.addCookie(cookie)응답에 쿠키 추가
request.getCookies()요청에 포함된 쿠키 배열 읽기

5-3. Cookie 수명

설정의미
setMaxAge(60 * 60)1시간 유지
setMaxAge(60 * 60 * 24 * 7)7일 유지
setMaxAge(0)쿠키 삭제
MaxAge 미설정브라우저 세션 쿠키로 동작할 수 있음
Cookie에는 비밀번호를 저장하지 않는다. “아이디 저장”은 가능하지만 “비밀번호 저장”은 수업 예제로도 피하는 것이 좋다. 로그인 인증 자체는 일반적으로 session을 사용하고, 쿠키는 보조 정보 저장에 사용한다.

6. localStorage

6-1. localStorage란?

localStorage는 브라우저가 제공하는 저장소이다. JavaScript로 값을 저장하고 읽을 수 있으며, 저장한 데이터는 같은 출처(origin) 안에서 유지된다. 서버로 자동 전송되지 않는다는 점이 Cookie와 가장 큰 차이다.

저장

localStorage.setItem("theme", "dark");

읽기

localStorage.getItem("theme");

삭제

localStorage.removeItem("theme");

6-2. Cookie와 localStorage 차이

구분CookielocalStorage
서버 자동 전송자동 전송됨자동 전송 안 됨
사용 언어Servlet/JSP, JavaScriptJavaScript
사용 예아이디 저장, 세션 ID다크 모드, 닉네임, 임시 입력값
문자열 저장기본적으로 문자열문자열. 객체는 JSON 변환 필요
localStorage 값은 서버가 자동으로 모른다. 서버가 알아야 한다면 form의 hidden input에 넣거나 fetch/AJAX로 직접 전송해야 한다.

6-3. sessionStorage는 무엇인가?

sessionStorage는 localStorage와 비슷하지만 브라우저 탭 단위로 유지된다. 탭을 닫으면 데이터가 사라진다. 이번 수업의 중심은 localStorage지만, 비교용으로 함께 알아두면 좋다.

7. 흐름으로 이해하기

7-1. 로그인 성공: session + redirect

login.jspPOST /login LoginServletsession 저장 redirect /home.jsp새 요청 home.jsp에서 session 확인

redirect를 하면 request는 사라진다. 그런데 session은 사용자 단위로 유지되기 때문에 home.jsp에서 로그인 상태를 확인할 수 있다.

7-2. 로그인 실패: request + forward

login.jspPOST /login LoginServletrequest에 error 저장 forward /login.jsp같은 요청 login.jsp에서 error 출력

로그인 실패 메시지는 한 번만 보여주면 되므로 session에 저장할 필요가 없다. request에 저장하고 forward하면 된다.

7-3. 아이디 저장: cookie

아이디 저장 체크로그인 요청 서버가 Cookie 생성응답 브라우저 저장다음 접속 쿠키 읽고 input에 표시

7-4. 화면 설정 저장: localStorage

다크 모드 클릭JS 실행 localStorage 저장새로고침 JS가 다시 읽어서 적용

이 과정은 Tomcat 서버에 요청하지 않아도 브라우저 안에서 처리할 수 있다.

2단계: LoginServlet.java

package com.example.auth;

import java.io.IOException;

import jakarta.servlet.ServletException;
import jakarta.servlet.annotation.WebServlet;
import jakarta.servlet.http.HttpServlet;
import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;
import jakarta.servlet.http.HttpSession;

@WebServlet("/login")
public class LoginServlet extends HttpServlet {

    @Override
    protected void doPost(HttpServletRequest request, HttpServletResponse response)
            throws ServletException, IOException {

        request.setCharacterEncoding("UTF-8");

        String id = request.getParameter("id");
        String pw = request.getParameter("pw");

        // DB 연결 전이므로 임시 로그인 조건으로 실습한다.
        if ("admin".equals(id) && "1234".equals(pw)) {
            HttpSession session = request.getSession();
            session.setAttribute("loginId", id);
            session.setMaxInactiveInterval(60 * 30); // 30분

            // 성공 후 새 요청으로 메인 화면 이동
            response.sendRedirect(request.getContextPath() + "/home.jsp");
        } else {
            // 실패 메시지는 이번 요청에서만 필요하므로 request에 저장
            request.setAttribute("error", "아이디 또는 비밀번호가 틀렸습니다.");
            request.getRequestDispatcher("/login.jsp").forward(request, response);
        }
    }
}

4단계: LogoutServlet.java

package com.example.auth;

import java.io.IOException;

import jakarta.servlet.ServletException;
import jakarta.servlet.annotation.WebServlet;
import jakarta.servlet.http.HttpServlet;
import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;
import jakarta.servlet.http.HttpSession;

@WebServlet("/logout")
public class LogoutServlet extends HttpServlet {

    @Override
    protected void doGet(HttpServletRequest request, HttpServletResponse response)
            throws ServletException, IOException {

        HttpSession session = request.getSession(false);

        if (session != null) {
            session.invalidate();
        }

        response.sendRedirect(request.getContextPath() + "/login.jsp");
    }
}
실행 순서: /login.jsp 접속 → 아이디 admin, 비밀번호 1234 입력 → 로그인 성공 → home.jsp 이동 → 로그아웃 클릭.

String cookiePath = request.getContextPath().isEmpty()

rememberCookie.setHttpOnly(true);

3단계: LoginServlet 전체 예시

package com.example.auth;

import java.io.IOException;

import jakarta.servlet.ServletException;
import jakarta.servlet.annotation.WebServlet;
import jakarta.servlet.http.Cookie;
import jakarta.servlet.http.HttpServlet;
import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;
import jakarta.servlet.http.HttpSession;

@WebServlet("/login")
public class LoginServlet extends HttpServlet {

    @Override
    protected void doPost(HttpServletRequest request, HttpServletResponse response)
            throws ServletException, IOException {

        request.setCharacterEncoding("UTF-8");

        String id = request.getParameter("id");
        String pw = request.getParameter("pw");
        String remember = request.getParameter("remember");

        // 아이디 저장 쿠키 처리
        String cookiePath = request.getContextPath().isEmpty()
                ? "/"
                : request.getContextPath();

        Cookie rememberCookie = new Cookie("rememberId", id);
        rememberCookie.setPath(cookiePath);
        rememberCookie.setHttpOnly(true);

        if ("on".equals(remember)) {
            rememberCookie.setMaxAge(60 * 60 * 24 * 7);
        } else {
            rememberCookie.setMaxAge(0);
        }

        response.addCookie(rememberCookie);

        // DB 연결 전이므로 임시 로그인 조건으로 실습한다.
        if ("admin".equals(id) && "1234".equals(pw)) {
            HttpSession session = request.getSession();
            session.setAttribute("loginId", id);

            response.sendRedirect(request.getContextPath() + "/home.jsp");
        } else {
            request.setAttribute("error", "아이디 또는 비밀번호가 틀렸습니다.");
            request.getRequestDispatcher("/login.jsp").forward(request, response);
        }
    }
}

4단계: 쿠키 읽기 기본 패턴

        }
    }
}

9. 자주 나는 오류와 주의점

9-1. Tomcat 11에서 javax import 오류

증상: javax.servlet.http.HttpSession, javax.servlet.http.Cookie를 찾을 수 없다고 나온다.

Tomcat 11은 Jakarta 계열이므로 import를 다음처럼 사용한다.

import jakarta.servlet.http.HttpSession;
import jakarta.servlet.http.Cookie;

수업에서 Tomcat 9를 쓰는 경우에는 반대로 javax.servlet.*를 사용한다.

9-2. 로그인 성공했는데 home.jsp에서 로그인 상태가 안 보임

  • session.setAttribute("loginId", id)를 했는지 확인한다.
  • 읽을 때 이름이 같은지 확인한다. 저장 이름이 loginId이면 읽을 때도 loginId여야 한다.
  • 로그인 후 같은 브라우저에서 접속하고 있는지 확인한다.
  • 개발자 도구에서 쿠키에 JSESSIONID가 있는지 확인한다.

9-3. redirect 후 request 값이 사라짐

정상이다. redirect는 새 요청이므로 request 데이터가 유지되지 않는다. redirect 후에도 필요한 값은 session에 저장해야 한다.

9-4. 쿠키가 삭제되지 않음

쿠키 삭제는 같은 이름, 같은 path로 MaxAge를 0으로 보내야 한다.

Cookie c = new Cookie("rememberId", "");
c.setPath(request.getContextPath().isEmpty() ? "/" : request.getContextPath());
c.setMaxAge(0);
response.addCookie(c);

9-5. localStorage 값이 서버에서 안 보임

localStorage는 request에 자동 포함되지 않는다. form hidden input, fetch, AJAX 등으로 직접 보내야 한다.

9-6. 보안 주의

  • 비밀번호를 cookie나 localStorage에 저장하지 않는다.
  • 로그인 상태는 서버 session으로 관리하는 것이 입문 단계에서는 가장 안전하고 이해하기 쉽다.
  • 중요 쿠키는 HttpOnly, Secure 설정을 고려한다.
  • localStorage는 JavaScript에서 읽을 수 있으므로 민감한 인증 정보 저장에 부적합하다.

10. 확인 문제

개념 확인

  1. redirect 후 request에 저장한 값은 유지되는가?
  2. 로그인 성공 후 사용자 아이디를 저장하기에 적절한 저장소는 무엇인가?
  3. 아이디 저장 체크박스에 적절한 저장소는 무엇인가?
  4. 다크 모드 설정처럼 서버가 몰라도 되는 값은 어디에 저장하기 좋은가?
  5. localStorage 값은 서버로 자동 전송되는가?
  6. 쿠키를 삭제할 때 사용하는 MaxAge 값은 무엇인가?
  7. getSession(false)는 언제 유용한가?

정답

1. 유지되지 않는다. redirect는 새 요청이다.

2. session.

3. cookie가 적절하다. 단, 비밀번호는 저장하지 않는다.

4. localStorage.

5. 자동 전송되지 않는다.

6. setMaxAge(0).

7. 기존 세션이 있는지만 확인하고, 없을 때 새 세션을 만들고 싶지 않을 때.
