2. Tomcat이란?

Tomcat은 Java 웹 애플리케이션을 실행해주는 서버 프로그램입니다. 브라우저는 HTML, CSS, JavaScript는 직접 해석할 수 있지만, JSP나 Servlet 같은 Java 코드는 직접 실행하지 못합니다.

그래서 중간에서 Java 웹 코드를 실행하고, 그 결과를 HTML로 만들어 브라우저에게 보내주는 프로그램이 필요합니다. 그 대표적인 프로그램이 Apache Tomcat입니다.

1브라우저가 HTTP 요청을 보낸다.
2Tomcat이 요청을 받는다.
3Tomcat이 JSP 또는 Servlet을 실행한다.
4Java 코드 실행 결과로 HTML 응답을 만든다.
5브라우저가 HTML을 받아 화면에 출력한다.
한 문장 정리

Tomcat = Java 웹 서버 + Servlet/JSP 실행기

Tomcat이 필요한 이유

HTML 파일은 브라우저가 바로 열 수 있습니다. 하지만 JSP 파일은 Java 코드가 포함될 수 있기 때문에 브라우저가 직접 실행할 수 없습니다. 

index.html
→ 브라우저가 직접 해석 가능

index.jsp
→ Java 코드가 포함될 수 있음
→ 브라우저가 직접 실행 불가능
→ Tomcat이 실행해야 함

예를 들어 JSP 파일에 다음 코드가 있다고 가정합니다.

<%
    String name = "홍길동";
%>

<h1><%= name %>님 안녕하세요</h1>

브라우저는 <% %> 안의 Java 코드를 알지 못합니다. Tomcat이 JSP를 실행한 뒤, 최종 결과인 HTML을 만들어 브라우저에게 응답합니다. 

<h1>홍길동님 안녕하세요</h1>

Tomcat의 역할

역할 설명
요청 받기 브라우저가 보낸 HTTP 요청을 받는다.
주소 분석 /hello, /login, /index.jsp 같은 요청 주소를 확인한다.
JSP 실행 JSP를 실행하고 HTML 결과를 만든다.
Servlet 실행 요청 주소와 연결된 Servlet 클래스를 실행한다.
응답 생성 HTML, 페이지 이동, 오류 코드 등을 브라우저로 보낸다.
포트 관리 보통 8080 포트에서 요청을 기다린다.

Apache HTTP Server와 Tomcat 차이

구분 주요 역할 예시
Apache HTTP Server 정적 웹 서버 HTML, CSS, JS, 이미지 제공
Apache Tomcat Java 웹 서버 JSP, Servlet 실행

3. Tomcat 버전 선택

수업에서 가장 많이 헷갈리는 부분은 Tomcat 9와 Tomcat 10 이상에서 사용하는 Servlet 패키지명이 다르다는 점입니다.

구분 패키지명 예시 import 수업 기준
Tomcat 9 javax.servlet.* import javax.servlet.http.HttpServlet; 예전 JSP/Servlet 교재에서 많이 사용
Tomcat 10.1 이상 jakarta.servlet.* import jakarta.servlet.http.HttpServlet; 최신 Jakarta EE 기준
버전 선택 공식
수업 코드가 javax.servlet.* 이면
→ Tomcat 9 사용

수업 코드가 jakarta.servlet.* 이면
→ Tomcat 10.1 이상 사용

가장 중요한 것은 JDK, Tomcat, Servlet import 버전이 서로 맞아야 한다는 점입니다. 버전이 맞지 않으면 import 오류나 실행 오류가 발생할 수 있습니다.

5. Windows 기준 Tomcat 설치 방법

5-1. JDK 설치 확인

명령 프롬프트를 열고 다음 명령어를 입력합니다.

java -version
javac -version

정상적으로 버전이 나오면 Java 설치가 된 것입니다.

java version "17.0.x"
javac 17.0.x
주의

java 또는 javac 명령어를 찾을 수 없다고 나오면 JDK 설치 또는 환경변수 설정을 확인해야 합니다.

5-2. JAVA_HOME 설정

Tomcat은 Java로 실행되므로 JDK 위치를 알아야 합니다. 이때 사용하는 환경변수가 JAVA_HOME입니다.

예를 들어 JDK가 다음 위치에 설치되어 있다면,

C:\Program Files\Java\jdk-17

환경변수는 다음처럼 설정합니다.

JAVA_HOME = C:\Program Files\Java\jdk-17

Path에는 다음 값을 추가합니다.

%JAVA_HOME%\bin
구분 예시
올바른 JAVA_HOME C:\Program Files\Java\jdk-17
잘못된 JAVA_HOME C:\Program Files\Java\jdk-17\bin

설정 확인은 다음 명령어로 합니다.

echo %JAVA_HOME%
java -version
javac -version

5-3. Tomcat 다운로드

Apache Tomcat 공식 다운로드 페이지에서 수업 버전에 맞는 Tomcat을 내려받습니다. 초보 수업에서는 보통 설치 프로그램보다 zip 파일 방식이 이해하기 쉽습니다.

수업 기준 다운로드 예시
Tomcat 9 apache-tomcat-9.x.x-windows-x64.zip
Tomcat 10.1 apache-tomcat-10.1.x-windows-x64.zip
zip 방식의 장점
  • 압축만 풀면 설치가 끝난다.
  • 폴더 구조를 직접 확인할 수 있다.
  • Eclipse에 연결하기 쉽다.
  • 삭제할 때 폴더만 지우면 된다.

5-4. Tomcat 압축 해제

다운로드한 zip 파일을 원하는 위치에 압축 해제합니다.

추천 위치:

C:\dev\apache-tomcat-9.0.x
C:\dev\apache-tomcat-10.1.x

비추천 위치:

바탕화면
다운로드 폴더
한글 경로가 포함된 폴더
너무 깊은 폴더
용어 정리

Tomcat을 압축 해제한 최상위 폴더를 보통 Tomcat 설치 폴더 또는 CATALINA_HOME이라고 부릅니다.

6. Tomcat 폴더 구조

압축을 풀면 다음과 같은 폴더가 보입니다.

apache-tomcat-9.0.x
 ├─ bin
 ├─ conf
 ├─ lib
 ├─ logs
 ├─ temp
 ├─ webapps
 └─ work
폴더 의미
bin Tomcat 실행, 종료 파일이 들어 있다. startup.bat, shutdown.bat
conf 서버 설정 파일이 들어 있다. 포트 번호를 바꿀 때 server.xml을 수정한다.
lib Tomcat 실행에 필요한 라이브러리 파일이 들어 있다.
logs 실행 로그와 오류 로그가 저장된다.
webapps 웹 프로젝트가 배포되는 위치이다.
work JSP가 Servlet으로 변환되어 임시 저장되는 위치이다.
수업 초반에 특히 중요한 폴더

bin, conf, webapps, logs, work

7. Tomcat 실행과 종료

7-1. Tomcat 실행

Tomcat 설치 폴더 안의 bin 폴더로 이동합니다.

C:\dev\apache-tomcat-9.0.x\bin

여기서 startup.bat을 실행하면 Tomcat이 시작됩니다.

cd C:\dev\apache-tomcat-9.0.x\bin
startup.bat

7-2. 실행 확인

브라우저에서 다음 주소로 접속합니다.

http://localhost:8080
부분 의미
http HTTP 방식으로 요청한다.
localhost 내 컴퓨터를 의미한다.
8080 Tomcat이 요청을 기다리는 기본 포트 번호이다.
주소 해석

http://localhost:8080은 “내 컴퓨터에서 실행 중인 Tomcat 서버의 8080번 포트로 요청한다”는 뜻입니다.

7-3. Tomcat 종료

Tomcat을 종료할 때는 shutdown.bat을 실행합니다.

cd C:\dev\apache-tomcat-9.0.x\bin
shutdown.bat

7-4. 8080 포트 변경

다른 프로그램이 이미 8080 포트를 사용 중이면 Tomcat이 실행되지 않을 수 있습니다. 이때는 conf/server.xml에서 포트 번호를 변경합니다.

기존 설정:

<Connector port="8080" protocol="HTTP/1.1"
           connectionTimeout="20000"
           redirectPort="8443" />

변경 예시:

<Connector port="8090" protocol="HTTP/1.1"
           connectionTimeout="20000"
           redirectPort="8443" />

변경 후에는 다음 주소로 접속합니다.

http://localhost:8090

8. Eclipse에 Tomcat 등록하기

Tomcat을 설치한 뒤에는 Eclipse에서 사용할 수 있도록 서버 런타임으로 등록해야 합니다.

8-1. Tomcat Runtime 등록

  1. Eclipse 실행
  2. Window → Preferences 이동
  3. Server → Runtime Environments 또는 Installed Runtimes 선택
  4. Add 클릭
  5. Apache → Apache Tomcat v9.0 또는 Apache Tomcat v10.1 선택
  6. Tomcat 설치 폴더 선택
  7. JDK 선택 후 Finish
폴더 선택 주의

Tomcat 설치 폴더를 선택해야 하며, bin 폴더를 선택하면 안 됩니다.

잘못된 선택:
C:\dev\apache-tomcat-9.0.x\bin

올바른 선택:
C:\dev\apache-tomcat-9.0.x

8-2. Eclipse에서 서버 만들기

  1. Servers 탭 열기
  2. No servers are available 또는 빈 영역에서 새 서버 생성
  3. Apache → Tomcat v9.0 Server 또는 Tomcat v10.1 Server 선택
  4. 실행할 프로젝트 추가
  5. Finish
  6. 서버를 Start하여 실행 확인

8-3. Dynamic Web Project와 연결

웹 프로젝트를 만들 때는 Target Runtime으로 Tomcat을 선택합니다.

File
 → New
 → Dynamic Web Project
 → Project name 입력
 → Target Runtime: Apache Tomcat v9.0 또는 v10.1 선택
Target Runtime이란?

이 프로젝트를 어떤 서버에서 실행할지 정하는 설정입니다. JSP와 Servlet 프로젝트라면 Tomcat Runtime을 연결해야 합니다.

9. 첫 JSP 실행

프로젝트 이름이 MyWeb이고, index.jsp 파일이 있다고 가정합니다.

<%@ page contentType="text/html; charset=UTF-8" %>
<!DOCTYPE html>
<html>
<head>
    <title>첫 JSP</title>
</head>
<body>
    <h1>Tomcat 실행 성공</h1>
</body>
</html>

브라우저에서 다음 주소로 접속합니다.

http://localhost:8080/MyWeb/index.jsp
1브라우저가 /MyWeb/index.jsp 요청
2Tomcat이 8080 포트에서 요청을 받음
3Tomcat이 MyWeb 프로젝트를 찾음
4index.jsp 실행
5HTML 응답 생성
6브라우저에 화면 출력
핵심

사용자는 JSP 원본 코드를 보는 것이 아니라, Tomcat이 JSP를 실행해서 만든 HTML 결과를 봅니다.

10. 요청 request와 응답 response

웹 수업의 가장 기본이 되는 개념은 요청과 응답입니다.

1사용자가 주소 입력, 링크 클릭, form 제출
2브라우저가 서버로 request 전송
3Tomcat이 요청을 받아 JSP 또는 Servlet 실행
4서버가 response 생성
5브라우저가 응답을 화면에 출력

request란?

request는 브라우저가 서버에게 보내는 요청 정보입니다.

  • 요청 주소
  • 요청 방식 GET 또는 POST
  • 사용자가 입력한 값
  • 브라우저 정보
  • 쿠키 정보

response란?

response는 서버가 브라우저에게 돌려주는 결과입니다.

  • HTML
  • CSS, JavaScript, 이미지
  • 페이지 이동 명령
  • 상태 코드
방향 기억하기
request  = 브라우저 → 서버
response = 서버 → 브라우저

11. GET과 POST

11-1. GET 요청

GET은 주로 조회할 때 사용합니다. 데이터가 주소 뒤에 붙습니다.

http://localhost:8080/MyWeb/hello.jsp?name=hong

? 뒤에 붙는 값을 query string이라고 합니다.

name=hong

JSP에서는 다음처럼 값을 받을 수 있습니다.

<%@ page contentType="text/html; charset=UTF-8" %>
<%
    String name = request.getParameter("name");
%>

<h1><%= name %>님 안녕하세요</h1>

11-2. POST 요청

POST는 주로 사용자가 입력한 데이터를 서버로 보낼 때 사용합니다.

<form action="result.jsp" method="post">
    <input type="text" name="name">
    <button type="submit">전송</button>
</form>

result.jsp에서는 다음처럼 값을 받습니다.

<%@ page contentType="text/html; charset=UTF-8" %>
<%
    request.setCharacterEncoding("UTF-8");
    String name = request.getParameter("name");
%>

<h1><%= name %>님 환영합니다.</h1>

11-3. GET과 POST 비교

구분 GET POST
주 목적 조회 등록, 전송
데이터 위치 URL 뒤 query string 요청 body
주소창 노출 보임 URL에는 직접 보이지 않음
사용 예 검색, 목록, 상세보기 로그인, 회원가입, 글쓰기
form method method="get" method="post"
암기 공식
GET  = 주소에 값을 붙여서 요청
POST = 요청 몸체에 값을 담아서 요청

11-4. form 태그 핵심

사용자 입력값을 서버로 보내려면 form 태그를 사용합니다.

<form action="result.jsp" method="get">
    아이디: <input type="text" name="id">
    비밀번호: <input type="password" name="pw">
    <button type="submit">로그인</button>
</form>
속성 의미
action 요청을 보낼 주소
method 요청 방식 GET 또는 POST
name 서버에서 값을 꺼낼 때 사용하는 이름
가장 중요한 연결
<input type="text" name="id">
             ↓
request.getParameter("id")

11-5. 한글 깨짐 처리

POST 방식으로 한글을 보낼 때는 값을 꺼내기 전에 인코딩을 설정합니다.

request.setCharacterEncoding("UTF-8");

String name = request.getParameter("name");
순서 주의

request.setCharacterEncoding("UTF-8")은 반드시 request.getParameter()보다 먼저 작성해야 합니다.

12. Servlet으로 요청 처리하기

JSP가 화면 중심이라면, Servlet은 요청 처리 중심입니다. Servlet은 브라우저의 요청을 받아 Java 코드로 처리하고 응답을 만듭니다.

역할 구분
JSP     : 화면 출력
Servlet : 요청 처리

12-1. Servlet 기본 구조

Tomcat 10.1 이상 기준 예시입니다.

import java.io.IOException;
import java.io.PrintWriter;

import jakarta.servlet.ServletException;
import jakarta.servlet.annotation.WebServlet;
import jakarta.servlet.http.HttpServlet;
import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;

@WebServlet("/hello")
public class HelloServlet extends HttpServlet {

    @Override
    protected void doGet(HttpServletRequest request, HttpServletResponse response)
            throws ServletException, IOException {

        response.setContentType("text/html; charset=UTF-8");

        PrintWriter out = response.getWriter();
        out.println("<h1>Hello Servlet</h1>");
    }
}

Tomcat 9를 사용한다면 import가 다음처럼 바뀝니다.

import javax.servlet.ServletException;
import javax.servlet.annotation.WebServlet;
import javax.servlet.http.HttpServlet;
import javax.servlet.http.HttpServletRequest;
import javax.servlet.http.HttpServletResponse;

12-2. Servlet 요청 주소

Servlet 위에 작성한 @WebServlet은 요청 주소를 연결합니다.

@WebServlet("/hello")

이 뜻은 다음과 같습니다.

/hello 주소로 요청이 오면 이 Servlet을 실행한다.

브라우저에서 다음 주소로 접속하면 HelloServlet의 doGet()이 실행됩니다.

http://localhost:8080/MyWeb/hello

12-3. doGet과 doPost

메서드 실행 상황 예시
doGet() GET 요청이 들어왔을 때 실행 주소 직접 입력, 링크 클릭, 검색
doPost() POST 요청이 들어왔을 때 실행 로그인, 회원가입, 글쓰기 form 제출

12-4. Servlet에서 form 값 받기

입력 화면입니다.

<form action="login" method="post">
    아이디: <input type="text" name="id">
    비밀번호: <input type="password" name="pw">
    <button type="submit">로그인</button>
</form>

Servlet에서는 다음처럼 값을 받습니다.

@WebServlet("/login")
public class LoginServlet extends HttpServlet {

    @Override
    protected void doPost(HttpServletRequest request, HttpServletResponse response)
            throws ServletException, IOException {

        request.setCharacterEncoding("UTF-8");

        String id = request.getParameter("id");
        String pw = request.getParameter("pw");

        System.out.println("id = " + id);
        System.out.println("pw = " + pw);
    }
}
연결 관계
form action="login"
        ↓
@WebServlet("/login")

input name="id"
        ↓
request.getParameter("id")

12-5. Servlet에서 직접 응답 보내기

Servlet에서 직접 HTML을 만들어 브라우저로 보낼 수 있습니다.

response.setContentType("text/html; charset=UTF-8");

PrintWriter out = response.getWriter();
out.println("<html>");
out.println("<body>");
out.println("<h1>로그인 결과</h1>");
out.println("<p>로그인 요청이 처리되었습니다.</p>");
out.println("</body>");
out.println("</html>");
실무 구조로 가면?

Servlet 안에 HTML을 많이 쓰면 코드가 복잡해집니다. 그래서 보통 Servlet은 요청을 처리하고, JSP는 화면을 출력하도록 역할을 나눕니다.

13. forward와 redirect

13-1. forward

forward는 Servlet에서 처리한 결과를 JSP로 넘길 때 사용합니다. 서버 내부에서 이동하기 때문에 request 데이터가 유지됩니다. 

request.setAttribute("message", "로그인 성공");

request.getRequestDispatcher("/result.jsp")
       .forward(request, response);

result.jsp에서는 다음처럼 출력할 수 있습니다.

<h1>${message}</h1>
forward 특징
  • 서버 내부에서 JSP로 이동한다.
  • 주소창이 바뀌지 않는다.
  • request에 담은 데이터가 유지된다.
  • 처리 결과를 화면에 보여줄 때 자주 사용한다.

13-2. redirect

redirect는 브라우저에게 다른 주소로 다시 요청하라고 지시하는 방식입니다. 

response.sendRedirect("main.jsp");
redirect 특징
  • 브라우저가 새 요청을 보낸다.
  • 주소창이 바뀐다.
  • 기존 request 데이터는 유지되지 않는다.
  • 작업 후 다른 페이지로 이동시킬 때 사용한다.

13-3. 비교

구분 forward redirect
이동 방식 서버 내부 이동 브라우저가 새 요청
주소창 바뀌지 않음 바뀜
request 데이터 유지됨 사라짐
주 사용 상황 결과를 JSP에 보여줄 때 다른 페이지로 이동시킬 때
암기 공식
forward  = request를 들고 서버 안에서 이동
redirect = 브라우저에게 다시 요청하라고 시킴

13-4. getParameter와 setAttribute 차이

메서드 용도 방향
request.getParameter() 사용자가 보낸 값을 받는다. 브라우저 → 서버
request.setAttribute() Servlet에서 JSP로 값을 전달한다. Servlet → JSP

14. 자주 나는 오류

14-1. 404 오류

404는 요청한 주소를 찾을 수 없다는 뜻입니다.

확인할 것
  • 프로젝트명이 맞는가?
  • JSP 파일명이 맞는가?
  • @WebServlet 주소가 맞는가?
  • 프로젝트가 Tomcat에 추가되어 있는가?
  • Tomcat 서버가 정상 실행 중인가?

14-2. 405 오류

405는 요청 방식이 맞지 않을 때 발생할 수 있습니다.

예시
  • GET으로 요청했는데 Servlet에 doGet()이 없다.
  • POST로 요청했는데 Servlet에 doPost()가 없다.
  • form의 method와 Servlet 메서드가 맞지 않는다.

14-3. 500 오류

500은 서버 내부 오류입니다. Java 코드 실행 중 예외가 발생했을 가능성이 큽니다.

확인할 것
  • Eclipse Console 오류 메시지
  • 오타
  • null 값 사용 여부
  • import 오류
  • Tomcat 버전과 Servlet import 불일치

14-4. Tomcat 실행 창이 바로 꺼짐

대부분 Java 환경변수 문제일 가능성이 높습니다.

java -version
javac -version
echo %JAVA_HOME%

14-5. 포트 충돌

8080 포트를 이미 다른 프로그램이 사용 중이면 Tomcat 실행이 실패할 수 있습니다.

conf/server.xml 열기
 → 8080 검색
 → 8090 등으로 변경
 → Tomcat 재시작

실습 5. Servlet으로 요청 처리하기

Tomcat 10 기준 Servlet 예시입니다. Tomcat 9를 사용하면 import를 javax.servlet.*로 바꿉니다.

import java.io.IOException;

import jakarta.servlet.ServletException;
import jakarta.servlet.annotation.WebServlet;
import jakarta.servlet.http.HttpServlet;
import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;

@WebServlet("/result")
public class ResultServlet extends HttpServlet {

    @Override
    protected void doPost(HttpServletRequest request, HttpServletResponse response)
            throws ServletException, IOException {

        request.setCharacterEncoding("UTF-8");

        String name = request.getParameter("name");

        request.setAttribute("name", name);

        request.getRequestDispatcher("/result.jsp")
               .forward(request, response);
    }
}

result.jsp

<%@ page contentType="text/html; charset=UTF-8" %>
<!DOCTYPE html>
<html>
<head>
    <title>결과 화면</title>
</head>
<body>
    <h1>결과 페이지</h1>
    <p>${name}님 환영합니다.</p>
</body>
</html>

16. 확인 문제

문제 1. Tomcat은 왜 필요한가?
답: 브라우저가 직접 실행할 수 없는 JSP와 Servlet 같은 Java 웹 코드를 실행하고, 그 결과를 HTML 응답으로 만들어 보내기 위해 필요하다.
문제 2. http://localhost:8080에서 localhost와 8080은 각각 무엇을 의미하는가?
답: localhost는 내 컴퓨터, 8080은 Tomcat이 요청을 기다리는 포트 번호를 의미한다.
문제 3. form 태그의 action, method, input의 name은 각각 무엇인가?
답: action은 요청 주소, method는 요청 방식, name은 서버에서 값을 꺼낼 때 사용하는 이름이다.
문제 4. request.getParameter("id")에서 "id"는 어디와 연결되는가?
답: HTML input 태그의 name="id"와 연결된다.
문제 5. forward와 redirect의 가장 큰 차이는 무엇인가?
답: forward는 서버 내부 이동이라 request가 유지되고 주소창이 바뀌지 않는다. redirect는 브라우저가 새 요청을 보내므로 request가 유지되지 않고 주소창이 바뀐다.

17. 최종 정리

이번 장 핵심 흐름
브라우저에서 주소 입력 또는 form 제출
        ↓
Tomcat이 요청을 받음
        ↓
JSP 또는 Servlet 실행
        ↓
request.getParameter()로 사용자 입력값 처리
        ↓
response 또는 JSP로 결과 생성
        ↓
브라우저에 HTML 응답

꼭 외울 것

  • Tomcat은 JSP와 Servlet을 실행하는 Java 웹 서버이다.
  • 브라우저는 서버에게 request를 보낸다.
  • 서버는 브라우저에게 response를 돌려준다.
  • JSP는 주로 화면을 출력한다.
  • Servlet은 주로 요청을 처리한다.
  • GET은 주소에 데이터가 붙는다.
  • POST는 요청 body에 데이터가 담긴다.
  • form의 action은 요청 주소이다.
  • form의 method는 요청 방식이다.
  • input의 name은 서버에서 값을 꺼낼 때 사용하는 이름이다.
  • request.getParameter()는 사용자가 보낸 값을 받는다.
  • request.setAttribute()는 JSP로 값을 전달한다.
  • forward는 request를 유지한다.
  • redirect는 새 요청이므로 request가 유지되지 않는다.
  • Tomcat 9는 javax.servlet.*, Tomcat 10 이상은 jakarta.servlet.*를 사용한다.
